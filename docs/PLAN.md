## Фаза 4 — Пайплайн медиа

1. **URL:** запуск `yt-dlp` с аргументами на извлечение аудио в временную папку (`std::env::temp_dir` + подпапка v2t); поддержка плейлиста (один элемент очереди → много выходных треков или разбиение на подзадачи — зафиксировать один подход в коде).
2. **Локальный файл:** при необходимости нормализация через `ffmpeg` (16 kHz, mono, WAV или MP3 — выбрать один формат для API).
3. Логирование stderr/stdout в лог-панель (обрезка длины строк).
4. Ошибки: таймауты, код возврата ≠ 0, файл не найден — понятное сообщение в UI.

**Критерий готовности:** из тестового URL и локального файла получается нормализованный аудиофайл на диске (промежуточный).

**Статус:** yt-dlp → ffmpeg → 16 kHz mono WAV в temp; таймауты и сообщения об ошибках; плейлисты — несколько `normalized_*.wav` на одну задачу очереди; в UI — событие `pipeline-log` (хвост stderr yt-dlp/ffmpeg).

## Фаза 5 — Транскрипция (API)

1. HTTP-клиент в Rust (`reqwest`): загрузка файла на endpoint совместимый с OpenAI **audio transcriptions** (multipart), настраиваемые `base_url`, `model`, заголовок авторизации.
2. Ответ → текст; запись в выходную папку с **шаблоном имени** (`{title}`, `{index}`, `{date}`, `{track}`, `{source}` и т.д.) + санитизация имён файлов Windows/macOS.
3. Опция «удалить аудио после успеха».
4. Большие файлы: если API ограничивает размер — задокументировать лимит; опционально нарезка через ffmpeg (Фаза 5.1 или отдельный этап).

**Статус:** `transcribe` (reqwest + rustls), `output_template` в Rust, команда `process_queue_item` (prepare → API → файл(ы), опционально удаление temp WAV), эмит прогресса; WAV выше ~22 MiB — нарезка ffmpeg на сегменты и склейка текста (`transcribe_wav_maybe_split`); команда `prepare_media_audio` сохранена для отладки.

**Критерий готовности:** end-to-end: один локальный файл → один `.txt` в выбранной папке.

## Фаза 7 — Локальная транскрипция (CLI Whisper, без модели в дистрибутиве)

**Принцип поставки:** в комплект **не входят файлы моделей** (сотни МБ–ГБ). В комплект входит только **CLI** (целевой вариант — **whisper.cpp** `main` / совместимый бинарник под Windows и macOS отдельными сборками), либо путь к нему задаётся в настройках (как у `ffmpeg` / `yt-dlp`).

1. **Режим транскрипции в настройках:** `HTTP API` (текущее поведение) **или** `Local Whisper (CLI)`.
2. **Выбор модели (только для локального режима):** фиксированный перечень поддерживаемых имен (например `tiny`, `base`, `small`, …) — в UI выпадающий список; в настройках хранится выбранное значение и **путь к каталогу моделей** (например `app_data_dir/v2t/models` по умолчанию).
3. **Процедура при старте задачи или по кнопке «Загрузить модель»:** если файла выбранной модели в каталоге нет — **скачивание по HTTPS** (HF `ggerganov/whisper.cpp`), **проверка SHA-1** (как в upstream `models/README.md`), событие **`model-download-progress`**; после успеха — CLI с `-m <path>`.
4. **Интеграция в пайплайн:** после `prepare_media_audio` — ветка: API как сейчас **или** subprocess к whisper.cpp (те же WAV/сегменты, что и для API), парсинг stdout/файла результата — по контракту выбранного CLI; отмена через существующий `CancellationToken` + kill процесса.
5. **Документация:** размеры моделей, офлайн-режим после первой загрузки, отдельная сеть для скачивания модели (capabilities).

**Статус:** режим **HTTP API / Local Whisper** в настройках; каталог моделей `whisper_catalog` + `model_download`; команды `list_whisper_models`, `default_whisper_models_dir`, `download_whisper_model`; ветка в `job.rs` после `prepare_media_audio`; `whisper_local` + нарезка WAV как у API; чеклист **whisper-cli** в `ReadinessPanel`; см. README.

**Критерий готовности:** переключение на локальный режим, выбор модели, первичная докачка, получение `.txt` без API key.

## Поставка `ffmpeg`, `yt-dlp` и опциональная авто-загрузка (рекомендация Алисы)

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| **A. Как сейчас: рядом с приложением / sidecar в релизе** | Предсказуемо, работает без интернета после копирования; меньше кода | Увеличение размера артефакта; нужно соблюдать лицензии сборок ffmpeg |
| **B. Опция «Скачать зависимости»** | Меньший установщик; пользователь сам решает | Нужен интернет; доверие к URL и checksum; обновления версий |

**Как правильно для v2t:** **A остаётся базовым** (портативная папка + README). **B — опциональная настройка**: по действию пользователя скачивать проверенные бинарники (статические сборки / официальные релизы) в **`app_data_dir`** (например `…/v2t/bin/`), зарегистрировать пути в настройках или использовать их приоритетно в «разрешении пути» после ручного override и sidecar. Не подменять системные `ffmpeg` без явного согласия пользователя.

*Лицензии:* учитывать условия выбранной сборки **ffmpeg** (LGPL/GPL в зависимости от билда); **yt-dlp** — обычно Unlicense; ссылки на источники — в README.
